  SECTION 5: KEY FINDINGS & RECOMMENDATIONS
═══════════════════════════════════════════════════════════════════════════════════

  ┌─────────────────────────────────────────────────────────────────────────────┐
  │  FINDING 1: YAML is Claude's native tongue for structured data            │
  │─────────────────────────────────────────────────────────────────────────────│
  │                                                                           │
  │  Every top-10 format either IS YAML or borrows YAML conventions.          │
  │  Claude's training corpus is saturated with YAML (K8s configs,            │
  │  GitHub Actions, Docker Compose, Ansible, etc). This means:               │
  │                                                                           │
  │  - Key-value pairs with colons are parsed near-perfectly                  │
  │  - Indentation-as-hierarchy is deeply internalized                        │
  │  - List items with dashes are unambiguous                                 │
  │  - YAML anchors/aliases are understood but add complexity                 │
  │                                                                           │
  │  Recommendation: Use YAML-like structure as the backbone.                 │
  └─────────────────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────────────────┐
  │  FINDING 2: Shorthand values > shorthand keys                             │
  │─────────────────────────────────────────────────────────────────────────────│
  │                                                                           │
  │  Abbreviating VALUES saves tokens without hurting parseability.            │
  │  Abbreviating KEYS hurts parseability more than it saves.                 │
  │                                                                           │
  │  Good: `spotify.auth_flow: PKCE, no server needed`                        │
  │  Good: `tech: Phaser3.88|TS|Vite|Supabase`                               │
  │  Bad:  `sp.af: PKCE`  (what is sp.af?)                                    │
  │  Bad:  `t: P3.88|TS|V|SB`  (cryptic)                                     │
  │                                                                           │
  │  Recommendation: Keep keys readable, compress values.                     │
  └─────────────────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────────────────┐
  │  FINDING 3: Status tags in brackets are universally understood            │
  │─────────────────────────────────────────────────────────────────────────────│
  │                                                                           │
  │  [PERFECT], [WIP], [SOLVED], [LOCKED], [NEVER], [ALWAYS] — these          │
  │  bracket-tag patterns appear across ALL top-performing formats.            │
  │  Claude parses them instantly because they mirror conventions from:        │
  │                                                                           │
  │  - Issue trackers: [BUG], [FEATURE], [CLOSED]                             │
  │  - Code comments: [TODO], [FIXME], [HACK]                                 │
  │  - Documentation: [DEPRECATED], [EXPERIMENTAL]                            │
  │                                                                           │
  │  Recommendation: Use [BRACKET_TAGS] for status and classification.        │
  └─────────────────────────────────────────────────────────────────────────────┘
